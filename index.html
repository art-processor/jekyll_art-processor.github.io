About ART

ART is a Variability Management Technique (VMT) using Frame Technology (FT).

Variability Management Techniques (VMT)

help developers manage the base code for families of system variants, e.g., a family of Linux versions running on various platforms, or financial systems used in different companies. VMTs organize and/or instrument the base code for ease of adaptation and reuse.

Frame technology (FT)

is a language-neutral (i.e., processes various languages) system that manufactures custom software from reusable, machine-adaptable building blocks, called frames. FT is used to reduce the time, effort, and errors involved in the design, construction, and evolution of large, complex software systems.

ART is here to help you automate common every day software engineering task through code generation and to help you eliminate human coding mistakes and achieve better code quality. However almost all our examples will talk about examples and usage in software engineering, ART has been proven to be useful in other aspects as well. In example for description to image generation (link, desc) and mass document handling (link, desc).

How it Works

ART is a simple text processor, reads text files line by line and outputs text files. If the current line in the currently read file is an ART command (#[cmd]), it will be processed, otherwise it will be emitted to the output.

Adapting input files

The ART Processor starts processing with the start file a.k.a specification (SPC) file. ART commands in the start file, and in each subsequently adapted file, are processed in the sequence in which they appear. Whenever the processor encounters an #adapt command in the currently processed file, processing of the current file is suspended and the Processor starts processing the given file. Once processing of the file is completed, the Processor resumes processing of the current file just after #adapt command. In that way the processing ends when the Processor reaches the end of the start file.

ART Story

XVCL era (2001 – 2012):

XVCL (XML-based Variant Configuration Language), an open source variability technique and tool was designed and built in 2001 with the collaboration of Stan Jarzabek, grand master of Code Clones and Software Product Lines and Paul Bassett, inventor of Frame Technology.

Software Product Lines (SPL)

Software product lines, or software product line development, refers to software engineering methods, tools and techniques for creating a collection of similar software systems from a shared set of software assets using a common means of production.

Carnegie Mellon Software Engineering Institute defines a software product line as

"A set of software-intensive systems that share a common, managed set of features satisfying the specific needs of a particular market segment or mission and that are developed from a common set of core assets in a prescribed way."

XVCL was developed in Software Engineering Lab at NUS (National University of Singapore). Initially, this work focused on variability management, to help domain engineers design generic, adaptable core assets, and to automated reuse-based product development. It is a uniform technique to manage variability in SPL core assets such as code, configurations, documentation, models, test cases, etc.

XVCL has been applied to manage variability in SPLs in a range of application domains (command & control systems, business applications, games for mobile phones, Web portals), using variety of programming technologies (Java, ASP, .NET, JEE). The technology became popular in research and industry projects as well. (See Publications section)

Some of the XVCL achievements

Case studies[5][6] were showing that using XVCL managed code fragments gives a 50 - 65% code base reduction, and makes code much easier to manage.
In the ASP.NET case study[6] the findings show an estimated eight-fold reduction of development effort and time-to-market in creating a new portal by reusing the designed XVCL frames.

VCL era (2012 – 2015):

As programming languages became easier to use and less strict in syntax. XML syntax seemed to become less popular and very restrictive compared to other options.

I've been fortunate to join an internship in 2012 January at NUS School of Computing. Before, I have been studying and working in Hungary at the University of Szeged, mostly working with on, and researching Software Code Quality and Software Code Maintenance.

During my internship in Singapore, working with Stan, we reimplemented XVCL as VCL (Variant Configuration Language) in an XML-free more user friendly and flexible syntax and conducted case studies with the outcome on Linux core code, Java Buffer Library and other projects as well. (See Publications section)

Some of the VCL achievements

Achieved 77.2% code reduction in the well-known Java Buffer Library by using VCL templates[2], making the code easier to read and enhance[x].
VCL is used today in industry for APS.NET web portal generation, has helped many projects which have won innovation and quality awards for development effort reduction and code stability improvements.

ART era (2015 – ):

ART (Adaptive Reuse Technology) is for research purpose. Built for you – for the community – to let you discover the potential behind this language and tool set and unlock the yet undiscovered potential in it.

ART is not exclusively for programming and/or source code generation. It has been used for generating image files, RTF documents, descriptions of products etc.
We are trying to keep an open mind. Most examples here will be about source code as that is what we are good at, and it is easier to enhance the things we understand and benefit from.

Some of the ART achievements

Achieved 61% code compression by refactoring cloned code instances into generic ART templates in Linux driver source code[1]. This allowed us to greatly enhance maintainability of the input driver code in Linux kernel code.

The best achievements are yet to come...
